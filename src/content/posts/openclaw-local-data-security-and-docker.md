很多人第一次接触 OpenClaw，最关心的不是“功能强不强”，而是更现实的问题：

**我的聊天记录、配置、密钥、文件，到底放在哪？会不会乱飞？**

再进一步一点，很多人会问：

**如果我用 Docker 安装，是不是就更安全了？能不能做到“完全隔离”？**

这篇文章不吹神话，也不搞吓唬人，就把这件事讲透：

- OpenClaw 的本地数据默认存在哪里
- 哪些数据只在本机，哪些数据会发给模型/外部服务
- OpenClaw 官方的安全模型到底是什么
- Docker 安装到底提升了什么安全性
- Docker 不能替你解决什么问题
- 如果你真在意本地数据安全，应该怎么配

## 一句话先说结论

如果你把 OpenClaw 跑在**自己控制的机器**上，并且：

- 只开放给自己或受信任的人使用
- 网关只监听本地回环地址或内网
- 开启认证
- 不乱给危险工具权限
- 需要时启用 Docker 沙箱

那么 OpenClaw 可以做到一个**相当实用的个人助理级安全基线**。

但也要说清楚：

**OpenClaw 不是“零信任神盒”，Docker 也不是“绝对保险箱”。**

它更像是这样一套模型：

1. **网关和宿主机是你的信任边界**
2. **本地数据默认存在你自己的磁盘上**
3. **模型需要看到的上下文，仍然可能被发送到你配置的模型提供商**
4. **Docker 能降低工具误操作和宿主机暴露面，但不能自动把一切都变成绝对安全**

如果你把它理解成“个人助理系统的风险收敛”，那是对的；如果理解成“扔进 Docker 就自动军事级保密”，那就想多了。

## 1. OpenClaw 的本地数据，默认放在哪里？

根据 OpenClaw 文档，容器化部署和常规部署最终都会在本地保留自己的状态目录与工作区。最关键的两个位置通常是：

- `~/.openclaw/`
- `~/.openclaw/workspace`

其中大概会包括这些内容：

### 1）配置和运行状态

比如：

- `openclaw.json` 配置
- 会话状态
- 设备配对状态
- 插件/工具相关配置
- 一部分日志与元数据

### 2）Agent 工作区

也就是 `workspace` 目录。

这里通常会有：

- `AGENTS.md`
- `SOUL.md`
- `USER.md`
- `MEMORY.md`
- `memory/YYYY-MM-DD.md`
- 你的脚本、草稿、导出的文件
- 代理在本地写下来的长期上下文

换句话说，**OpenClaw 的“记忆”和很多长期上下文，本质上就是本地文件**。

这有个好处：

你能看见、能备份、能审计、能删，不是一个完全黑盒的云端记忆仓。

但也有个代价：

**谁能读你的磁盘，谁就可能读到这些内容。**

所以本地数据安全的第一层，根本不是 Docker，而是：

- 你的操作系统账号是否安全
- 磁盘是否加密
- 目录权限是否正确
- 这台机器是不是被多人共用

## 2. “本地部署”不等于“所有数据永远不出本机”

这是很多人最容易误解的一点。

OpenClaw 跑在本地，表示：

- 网关和状态主要在你自己的机器上
- 配置、工作区、日志等默认落本地
- 你可以尽量把控制面收在自己手里

但只要你配置的是**远程模型提供商**，模型推理所需的上下文就依然可能会发到外部 API。

比如这些内容，可能根据你的实际使用被发送：

- 当前聊天内容
- 你让模型分析的片段
- 你贴进去的文本
- 工具执行结果摘要
- 某些嵌入/记忆检索所需内容
- TTS、网页抓取、搜索等外部能力涉及的数据

所以更准确的说法应该是：

**OpenClaw 可以把“控制权、存储权、审计权”尽量留在本地，但是否“完全离线”、是否“绝不出网”，取决于你配置的模型和工具链。**

如果你真的想追求更强的本地闭环，你应该优先考虑：

- 本地或私有化模型
- 关闭不必要的外部工具
- 最小化发送给模型的上下文
- 不把敏感文件内容直接喂给远程模型

## 3. OpenClaw 官方安全模型，核心假设是什么？

这个必须讲，不然很多安全讨论都容易跑偏。

OpenClaw 官方文档明确强调：

**它默认是“个人助理 trust model”，不是“敌对多租户隔离平台”。**

翻成人话就是：

- 适合你自己用，或者一小群**彼此信任**的人用
- 不适合把一个高权限 agent 暴露给一群互相不信任的人
- 如果要做多租户、强隔离，官方建议拆成不同网关、不同 OS 用户、甚至不同主机

这点特别关键。

因为很多人会天然地把它想成：

“这是个 bot，只要我开了账号隔离，应该就能像 SaaS 一样天然区分用户权限。”

但 OpenClaw 的实际定位不是这个。

它更接近：

**一个你自己掌控、能接消息、能跑工具、能调用模型的个人 AI 操作层。**

所以本地数据安全的真实前提是：

### 你得先保证“谁能碰这套系统”是可控的

重点包括：

- Gateway 不要裸奔公网
- 开认证，不要 `auth.mode="none"`
- 群聊/私聊入口做 pairing 或 allowlist
- 不要给开放群聊配高危工具权限
- 不要把个人环境和公司环境乱混在一个高权限 runtime 里

如果这一步没做好，后面再讨论 Docker 有多安全，基本都是扯淡。

## 4. Docker 安装，能提升哪些安全？

### 先说最核心的一句

**Docker 提升的是隔离性和误操作缓冲，不是自动加密，也不是万能防火墙。**

OpenClaw 文档里其实区分了两件事：

1. **整个 Gateway 跑在 Docker 里**
2. **Agent 工具执行跑在 Docker 沙箱里**

这两件事相关，但不是一回事。

### 4.1 整个 OpenClaw 跑在 Docker 里

这种模式下，Gateway 本体容器化，环境更可控，依赖更干净。

好处通常有这些：

#### 1）减少对宿主机环境的直接污染

你不用把一堆依赖直接装在宿主机上，升级、回滚和迁移也更容易。

#### 2）运行环境更可复制

容器镜像固定后，开发机、测试机、VPS 的行为更一致。

#### 3）能更容易配合额外的基础设施隔离

比如：

- 只暴露指定端口
- 配合反向代理
- 配合防火墙规则
- 用独立 volume 管理数据
- 限制容器能力和权限

#### 4）把“应用进程”和“宿主系统”隔开一层

哪怕应用里某个工具链出问题，理论上也比直接裸跑在宿主机上更容易收束影响面。

### 但别误会，Docker 版也有几个现实限制

#### 1）配置和工作区通常还是落在宿主机

官方 Docker 文档直接说明，容器化 Gateway 依然会把配置和工作区写到宿主机，比如：

- `~/.openclaw/`
- `~/.openclaw/workspace`

所以：

**Docker 不会自动把你的本地记忆、配置、密钥变成“宿主机不可见”。**

它只是改变运行方式，不会魔法般抹掉落盘事实。

#### 2）如果你挂了 bind mount，宿主路径照样暴露给容器

你挂多少、挂哪里，就暴露多少。

挂只读比读写好，但本质上仍然是把宿主目录交给容器看见。

#### 3）如果你把 Docker Socket 挂进去，风险会明显上升

尤其是为了启用更深层的 sandbox 或运维便利，把 `docker.sock` 暴露给容器时，要很谨慎。

因为能控制 Docker 的进程，往往就接近拥有宿主机级别的能力了。

OpenClaw 文档里也专门提醒：共享网络和 Docker 相关配置本身就是一个信任边界问题。

## 5. 更关键的是：Agent 工具沙箱

比“把 Gateway 装进 Docker”更值得关注的，其实是 **sandboxing**。

OpenClaw 支持把工具执行放进 Docker 容器里，从而减少 blast radius，也就是“模型瞎搞时能波及多大范围”。

官方文档讲得很直白：

- 如果沙箱关闭，工具就在宿主机跑
- 如果沙箱开启，`exec`、`read`、`write`、`edit`、`process` 等工具可以在隔离容器内运行
- 这不是绝对边界，但能显著限制文件系统和进程访问面

这件事非常重要。

因为很多真正危险的场景，不是“聊天记录泄露”，而是：

- 模型误删文件
- 跑错命令
- 扫到不该扫的目录
- 写坏工作区
- 读到宿主机上别的敏感文件

开启 sandbox 之后，风险控制会明显更好。

## 6. Docker 沙箱到底能做到什么程度？

如果你把 OpenClaw 的沙箱策略配得比较克制，它能做到这些实用层面的安全收益：

### 1）默认不给网络

文档里写得很明确：

**默认沙箱容器是没有网络的。**

这意味着很多“顺手往外发东西”的风险会被压住。

当然，一旦你主动打开网络，这层保护就打折了。

### 2）可以限制工作区访问方式

`sandbox.workspaceAccess` 支持：

- `none`
- `ro`
- `rw`

这玩意很实用。

比如：

- `none`：只给沙箱自己的空间，看不到你的 agent 工作区
- `ro`：只能读工作区，不能改
- `rw`：可以读写工作区

如果你只是让 agent 做分析、审阅、总结，很多时候 `ro` 就够了，没必要一上来就 `rw`。

### 3）可以阻止明显危险的挂载与网络模式

OpenClaw 对一些危险 Docker 配置本身就有防护，例如：

- 阻止危险 bind source
- 阻止 `network: "host"`
- 默认阻止 `container:<id>` 这类 namespace join

这说明它不是把 Docker 当摆设，而是真的在做“别让你手滑把自己炸了”的防呆。

### 4）可以把主会话和非主会话分开对待

`sandbox.mode` 支持：

- `off`
- `non-main`
- `all`

如果你想让平时主会话更顺手，但让群聊、外部来源、子任务更收敛，`non-main` 是个挺务实的折中。

## 7. 但 Docker 和沙箱，解决不了什么？

这一段反而最重要。

### 1）解决不了“你自己把敏感内容发给远程模型”

如果模型 API 是外部的，而你把敏感文件、密钥、聊天内容直接塞进上下文，Docker 并不能阻止这些内容被发送到模型服务。

**容器隔离的是本机执行环境，不是你主动发出去的网络请求语义。**

### 2）解决不了“入口权限配得一塌糊涂”

如果你：

- 对公网开放
- 没认证
- 群聊谁都能驱动工具
- 把高危 exec 开得很大

那就算装在 Docker 里，也只是“在一个盒子里被人狠狠干你”。

### 3）解决不了宿主机本身不安全

如果宿主机已经被入侵、多人共用、目录权限乱飞、磁盘不加密，那 OpenClaw 再怎么容器化，也只是建立在烂地基上的装修。

### 4）解决不了“elevated exec”这类主动绕过沙箱的能力

OpenClaw 文档明确说了：

- sandbox 不是万能的
- 某些显式允许的 host/elevated 执行会绕过沙箱

所以你只要给了高权限逃生通道，就别再幻想“我明明开了 Docker，所以绝对安全”。

## 8. 如果我用 Docker 安装，数据安全能做到什么水平？

一个比较诚实的回答是：

### 能做到的

如果你配置得当，Docker 安装的 OpenClaw 通常能达到下面这个级别：

- **本地数据主要保留在你控制的宿主机/卷里**
- **运行环境比直接裸装更整洁、更可控**
- **工具执行可以通过沙箱显著收缩影响面**
- **对“模型误操作本机文件系统”这类风险有明显缓解**
- **更容易做备份、迁移、版本回滚和环境审计**

### 做不到的

但它通常做不到这些神话级承诺：

- 不能保证“任何数据都不会出本机”
- 不能保证“容器内进程就绝对不可能影响宿主机”
- 不能自动实现“多用户敌对隔离”
- 不能替代磁盘加密、账户安全、网络隔离、密钥管理
- 不能替代你对模型提供商的数据边界判断

所以更靠谱的判断是：

**Docker 让 OpenClaw 变得更容易被收拾、更容易被约束、更不容易一把梭把宿主机搞脏，但它不是保险柜本身。**

## 9. 真正在意数据安全，建议怎么配？

如果你是个人开发者、小团队、或者自己在 VPS 上跑，我建议优先把下面这些事做好。

## 9.1 基础面先锁死

- Gateway 优先绑定 `loopback`
- 开启 token/password 认证
- 不要把控制面直接暴露公网
- 需要公网入口时，走明确的反向代理和防火墙
- 跑一遍 `openclaw security audit`

官方文档甚至直接给了安全审计命令：

```bash
openclaw security audit
openclaw security audit --deep
```

这个命令很适合做自查，别偷懒。

## 9.2 入口权限要收敛

- 私聊走 pairing 或 allowlist
- 群聊要求 mention 才触发
- 开放群里别给高危工具
- 不要让陌生人能碰到带 exec/fs/browser 的 agent

## 9.3 工具权限要最小化

- 不需要 `exec` 就关掉
- 不需要 elevated 就别开
- 文件系统尽量限制在 workspace
- 不需要联网的沙箱，就维持无网络

## 9.4 Docker 场景下尽量这样做

- 用独立 volume 管理 OpenClaw 数据
- 只挂必须的目录
- 能只读就别读写
- 不要随手把 `docker.sock` 暴露进去
- 容器和反向代理分网段/分权限
- 做好宿主机防火墙和系统更新

## 9.5 真敏感的数据，别指望“默认很安全”

如果是这些级别的数据：

- 公司核心源码
- 生产密钥
- 客户隐私数据
- 财务/法务文档

那建议你额外做这些事：

- 磁盘加密
- 独立机器或独立 VPS
- 独立 OS 用户
- 更严格的备份和审计策略
- 尽量使用更可控的模型与外部服务
- 把高敏任务和日常聊天 agent 分开跑

## 10. 我的建议：怎么理解“OpenClaw + Docker 的数据安全”

如果非要用一句话总结，我会这么说：

**OpenClaw 的本地数据安全，首先取决于宿主机和权限设计；Docker 能把风险明显收紧，但不是把风险抹掉。**

更直白一点：

- 你想要“自己的数据主要掌握在自己手里”——OpenClaw 可以做到
- 你想要“工具误操作不要直接扫穿整台机器”——Docker 沙箱很有用
- 你想要“随便开放给多人用还绝对隔离”——别做梦，得拆边界
- 你想要“远程模型还能看上下文，但又保证绝不出网”——这本身就自相矛盾

所以最靠谱的姿势不是迷信某一个技术名词，而是把整套链路看完整：

- 入口谁能进
- 网关怎么暴露
- 本地文件落在哪
- 模型请求发到哪
- 工具能碰什么
- Docker 沙箱怎么限权
- 宿主机本身安不安全

把这些都管住，OpenClaw 才会真靠谱。

## 最后一句

如果你只是想要一个简单判断：

**用 Docker 安装 OpenClaw，安全性通常会比直接高权限裸跑更好；但真正决定上限的，是你的部署边界、权限收敛和模型数据策略。**

别把 Docker 当护身符，把它当工具就对了。
